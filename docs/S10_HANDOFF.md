## Pipeline (one nightly call)

1. Pull ≤30 most recent drafts with `status IN ('published','metrics_collected','learned')` and a non-null `tweet_url`.
2. For each, call `publisher.get_post_metrics(tweet_url)`; on exception bump `source_health` and skip.
3. Compute `viral_score(...)` per draft → INSERT (OR REPLACE) into `post_metrics_timeseries` keyed on `posts.id` (looked up via `tweet_url`). Drafts with no `posts` row are skipped silently.
4. Take this batch's p30 / p70 of `viral_score`. Classify and dispatch:

| Case | trigger | action |
|---|---|---|
| ✅ winner | viral_score ≥ p70 (and > 0) | `apply_post_outcome(draft_id, "top")` → EMA α=0.3, clears cooling |
| ✅ dud | viral_score ≤ p30 | `apply_post_outcome(draft_id, "bottom")` → EMA α=0.1, bumps cooling miss; 3rd miss → 7-day pattern_cooling |
| ⚠️ mid | between | `apply_post_outcome(draft_id, "mid")` no-op bump usage |
| ❌ rejected high-scorer | already in `human_rejection_pool` (S13 wrote it) | reviewer does **not** re-process — read-only acknowledgment |

5. **Strategy weights**: aggregate avg viral_score by `topic_lane` over `window_days`. Lanes with `n < MIN_SAMPLE_FOR_WEIGHT_UPDATE (=5)` are skipped. Top third gets `+0.05`, bottom third `-0.05`; renormalize to sum=1.0. Snapshot before/after into `learning_log`.
6. **Channel B slop distillation**: group `technique_entries` by `hook_pattern`; patterns with `times_used_in_post ≥ 10` AND avg `success_score` in bottom 20% → INSERT into `slop_words (category='slop_pattern', source='reviewer_auto')` and re-render `voice/slop_words.md`.
7. **Stale drafts (§16.13 manual mode)**: drafts with `status='approved'` and `discord_reacted_at < now-7 days` → webhook alert (`src.alerting.alert` if present, else log). NOTE: we don't mutate status to `'stale'` because the CHECK constraint enumerates only 7 values; surfaced via `report.stale_drafts_alerted` instead. S1 can add the enum value later.

## 双通道沉淀

- **Channel A** — `strategy_weights` table + `learning_log` snapshot (topic-lane level).
- **Channel B** — `slop_words` table + `voice/slop_words.md` markdown (hook-pattern level, consumed by `src/guardrails.py` A-class rejector).

## Cron call (for S11 orchestrator)

```python
# every night at 00:00 CST (16:00 UTC the previous day)
# crontab.txt entry:
#   0 16 * * * cd /app && /usr/local/bin/python3 -m src.reviewer_cron >> logs/reviewer.log 2>&1

# src/reviewer_cron.py (S11-owned wrapper) example:
import asyncio
from src.reviewer import review_and_update_weights

async def main() -> None:
    report = await review_and_update_weights(window_days=7)
    print(f"reviewed={report.posts_reviewed} "
          f"updated_lanes={report.weights_updated} "
          f"slop={report.slop_words_added} "
          f"stale={report.stale_drafts_alerted} "
          f"took={report.duration_seconds}s")

if __name__ == "__main__":
    asyncio.run(main())
```

## Known caveats

- We rely on `posts.tweet_url` matching `drafts.tweet_url` to bridge timeseries; if S7 ever writes posts without backfilling URL, those drafts will be silently skipped from learning. Worth a future check in S5b self_monitor.
- `apply_post_outcome` is called with `draft_id`, which works because S13's `publisher_callback` writes `posts.id = draft_id` (or the writer/publisher pair pins `retrieval_log.post_id` to the draft). If that contract changes, the EMA bridge breaks.
- Stale-flag is alert-only until S1 extends the `drafts.status` enum.
